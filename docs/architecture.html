🧠 System Overview

AgentGuard implements a decentralized Human-in-the-Loop (HITL) governance layer for AI Agents. When an agent attempts a high-risk action, AgentGuard pauses execution and broadcasts a Request for Approval (RFA) to a decentralized network of verified human experts.

🤖

Governance Core

🌐

The high-level data flow is:

Python Agent → MCP Interceptor → Risk Classifier → P2P Broadcast → Human Pool → Identity Check → Approve/Reject → Resume/Abort Agent

🤖 Agent Layer (A2A Mesh)

The Agent Layer hosts multiple AI agents that collaborate via the A2A protocol. Each agent exposes an AgentCard at /.well-known/agent.json for discovery.

ComponentDescription
@haas_governed decoratorWraps Python functions to intercept critical tool calls
HaaSClientHTTP/WebSocket client communicating with haas-core
ContextPreserverSaves agent memory/stack during approval wait
ExceptionHandlerRaises HaaSApprovalDenied or HaaSTimeoutError

Python SDK Usage

from agentguard import haas

# The decorator intercepts the function call and routes it through HaaS governance
@haas.governed(tier="critical", pool="finance")
async def execute_wire_transfer(amount: float, recipient: str):
    # This code only runs if a human approves
    return banking_api.transfer(amount, recipient)

⚙️ Governance Core (MCP Server)

The central governance engine (TypeScript/Node.js) that intercepts, classifies, and orchestrates human approvals using the Model Context Protocol (MCP).

🔌

MCP Proxy Server

  • Wraps standard MCP tool servers
  • Transparent passthrough for Tier 1
  • Interception + pause for Tier 2-4
  • Returns input-required MCP state
🔌

MCP Interceptor

  • Intercepts tool_call payloads
  • Routes to Risk Classifier
  • Returns MCPToolResult

Core Configuration

import { HaaSCore } from '@agentguard/core';

const core = new HaaSCore({
  policies: './policies.yaml',
  redis: 'redis://localhost:6379',
  identity: ['linkedin', 'github', 'oidc'],
  telemetry: { endpoint: 'http://otel-collector:4318' }
});

await core.start();

🎯 Risk Classifier

Every tool call passes through the Risk Classifier, which determines the approval level required.

Tier Level Approval Threshold Examples
TIER 1 Low Auto-approve (logged) 0 signatures Read file, query data, generate report
TIER 2 Mid Single-Sig 1 signature / 1 pool Send email, update Jira, modify config
TIER 3 High Multi-Sig 2-3 signatures / 1 pool DB write, $1K-$100K transaction, deploy
TIER 4 Critical Cross-Pool Executive-Sig 2-3+ sigs / multiple pools Delete prod, $100K+ transfer, modify keys

Trust Requirements per Tier

TierMin SeniorityRequired Verification
LOWNoneNone
MIDAssociateVerified Workplace
HIGHSeniorWorkplace + Relevant Skill
CRITICALDirector / VPWorkplace + Skill + Multi-Factor

AIOps Feedback Loop

  • Auto-tuning: If Tier 3 approved 100 consecutive times → recommend downgrade to Tier 2
  • Escalation learning: Frequently rejected tool calls → recommend upgrading risk tier
  • Pattern recognition: Identifies common patterns and suggests policy optimizations

🔄 State Machine

The ApprovalStateMachine manages the lifecycle of every approval request, backed by Redis for crash recovery.

IDLE → PENDING → WAITING_FOR_HUMAN → PARTIAL_APPROVAL → APPROVED
WAITING_FOR_HUMAN → REJECTED
WAITING_FOR_HUMAN → TIMEOUT → ESCALATED
State Machine — Transition Diagram

🤝 Consensus Aggregator

Collects and validates cryptographic signatures from human reviewers, enforcing multi-sig thresholds.

1️⃣

Single-Sig (Tier 2)

First qualified human to verify and sign → action approved

2️⃣

Multi-Sig (Tier 3)

N unique signatures from ONE pool required (e.g., 2 from Finance)

3️⃣

Cross-Pool (Tier 4)

N signatures from MULTIPLE pools (e.g., Finance AND Security AND Legal)

🌐 P2P Network (LibP2P)

Decentralized broadcast layer using LibP2P GossipSub for Request for Approval (RFA) distribution.

PoolGossipSub TopicTier AccessTypical Validators
💰 Financehaas/finance3-4CFO, Finance Directors
🔒 Securityhaas/security4CISO, Security Engineers
🖥️ IT Operationshaas/itops2-3DevOps, SREs
📊 Datahaas/data2-3Data Engineers, DBAs
⚖️ Legalhaas/legal4Legal Counsel

Claim Pattern

1. RFA Broadcast
Request for Approval broadcast to relevant GossipSub topic(s)
2. Notification
Multiple humans receive notifications via push, desktop, or Slack
3. Claim
First human to "Claim" the task gets the identity challenge
4. Lock
Once claimed, other humans see "In Progress" — prevents race conditions
5. Fallback
If claimer fails identity check, task is re-released to pool

📜 Risk Policy Configuration

Policies are defined in YAML and loaded by the Risk Classifier on startup. Wildcards (*) match any suffix. 

# Risk Classification Policies
policies:
  # Critical: Large financial transactions
  - tool: "execute_transfer"
    conditions:
      - field: "args.amount"
        operator: ">"
        value: 100000
    tier: "CRITICAL"
    threshold: 3
    pools: ["finance", "security", "legal"]
    timeout: "1h"
    onRejection: "abort_agent"
    onTimeout: "escalate"

  # High: Moderate financial actions
  - tool: "execute_transfer"
    conditions:
      - field: "args.amount"
        operator: ">"
        value: 1000
    tier: "HIGH"
    threshold: 2
    pools: ["finance"]

  # Mid: Email communications
  - tool: "send_email"
    tier: "MID"
    threshold: 1
    pools: ["itops"]

  # Low: Read-only operations
  - tool: "read_*"
    tier: "LOW"
    autoApprove: true
